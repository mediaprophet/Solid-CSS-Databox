# ADR-0023 — Record awareness, access and correction capability

- **Status:** Adopted (capability) — the **CDR/APP compliance profile is a CANDIDATE only**, gated behind ADR-0015 (see Open sub-questions).
- **Date:** 2026-07-14
- **Decision owner:** DBX-02 (Hard lead)
- **Residual human review required:** legal-policy — the CDR 10-business-day clock, dashboard-disclosure rules and any APP timing are candidate configurations that require R-12/ADR-0015 applicability mapping and accountable human review before any compliance claim. privacy — existence-visibility and non-disclosure model.
- **Sources adjudicated:** R-14 (record awareness/access/correction); review-item #16 (appeal vs step-up — rejected as a single choice); architecture.md (Consumer record awareness); exchange-and-evidence.md (Immutability and correction); legal-review-cdr-data-awareness-and-correction.md.
- **Consumed by / blocks prompts:** DBX-14 (existence-vs-payload authorization, step-up denials), DBX-23 (correction connector + disposition).
- **Relates to:** ADR-0015 (compiled-policy gate for the compliance profile), ADR-0011 (append-only evidence/supersession), ADR-0012 (correction duties reuse the duty state machine), ADR-0016/0017 (source-system correction case + submission substrate).

## Context

A person can only exercise access and correction rights if they can first know a relevant record exists. legal-review-cdr-data-awareness-and-correction.md draws this as a chain: record awareness → authenticated access → correction request → acknowledgement → assessment → correction/linked-statement/refusal → downstream notification → complaint/review → durable evidence. R-14 makes record awareness a first-class Databox capability: an authenticated, program-local record and disclosure index with authorized metadata, current/superseded/disputed state, provenance, applicable policy and access/correction/complaint routes, with **existence visibility decided separately from payload access** so step-up and lawful non-disclosure work without side channels.

Two design errors must be avoided. review-item #16 rejects treating **appeal** and **step-up** as a single choice: step-up remedies *insufficient authentication*; appeal contests a *substantive decision*. And the legal-review file is explicit that the CDR Rules (7.9/7.10/7.14/7.15) and APP 10/12/13 supply a *candidate* profile, but "no build should claim legal compliance" until the corpus questions and human review (R-12) are resolved.

CSS 7.1.9 reality (DBX-01 §3): CSS already hides existence by returning **404 not 403** when an agent lacks Read (`PermissionBasedAuthorizer.reportAccessError`) — a reuse for the non-disclosure path. But structured reason codes and safe step-up responses are net-new (DBX-14). Existence-vs-payload as *separate authorization decisions* has no CSS equivalent and is new construction.

## Decision

**Provide an authenticated, program-local record & disclosure index (R-14; architecture.md `record-index`/`disclosure-view`).** For each item the consumer is authorized to see, it exposes only authorized metadata: opaque record id, record class, safe title/description; accountable organisation and source-system category; creation/effective/last-reviewed times; assertion vs opinion vs derived vs source; provenance and version/supersession status; current status (**current / superseded / disputed / correction-requested / corrected / statement-attached / access-restricted / retained-under-legal-obligation / tombstoned**); purposes where the profile makes them available; disclosure entries (what category, when, to whom) where required and lawful; applicable policy, retention and the assurance needed for payload access; and access/correction/statement/complaint/review actions. The index **complements** standard Solid containment — a conforming client still uses granted standard HTTP/LDP access; the index is an application projection, not a proprietary replacement (invariant 12).

**Existence visibility is decided SEPARATELY from payload access (R-14; architecture.md).** These are two distinct authorization decisions:

- A consumer MAY be permitted to *know an item exists* but require **step-up** authentication to *read the payload*.
- Where law or a valid safety/security exception requires it, the system MUST be able to **suppress even the existence** of an item — and MUST NOT leak it through counts, URIs, events, error-code differences, timing, or notifications. Program profiles carry an explicit `existenceVisibility` rule and a safe-refusal model. The existing 404-not-403 behaviour (DBX-01 §3) is reused so a payload denial does not reveal existence.

**Correction is an append-only governed exchange (R-14; exchange-and-evidence.md; legal-review file).** A correction request is a consumer submission (ADR-0017) linked to the target record and, where known, the specific field/assertion. On durable acceptance the Databox issues a **signed acknowledgement** carrying target, request digest, accepted time, applicable response profile, calculated due time and complaint route (acceptance does not imply the correction is true). Review records one or more **dispositions**: `corrected` (superseding version), `statement-associated` (conspicuously, machine-resolvably linked to the data), `partially-corrected`, `no-change` (reasons + complaint route), `more-information-required` (effect on deadline explicit), or `redirected`. Each correction sub-step is a **distinct duty** with its own target, due time, evidence and remedy (ADR-0012; rights-and-obligations.md: "accepting a request is not assessing it; assessing it is not correcting the source; signalling a prior recipient is not proof that corrected data was received"). The institutional source of truth MUST participate: the integration plane opens and reconciles a governed source-system case (ADR-0016/DBX-23) — a Databox-only annotation is insufficient if the institution keeps using the uncorrected source record. Where a prior recipient must be informed, the disclosure ledger creates **one independently-tracked duty per eligible recipient** (`dbx:notifyPriorRecipient`), with queued/attempted/accepted/failed states distinct and lawful exceptions applied; a notification hint is never proof the recipient's copy was corrected. All of this is append-only durable evidence (ADR-0011); history is never rewritten.

**Appeal and step-up are DISTINCT routes (review #16 rejected as a single choice).**

- **Step-up** remedies *insufficient authentication assurance* — the substantive decision might be favourable once the person authenticates at the required grade. A denial that is purely an assurance gap exposes a **step-up** route (new external authentication event, R-05/R-06 assurance model).
- **Appeal / review** contests a *substantive decision* (e.g. a `no-change` disposition or an access refusal on grounds other than assurance). It names a responsible body, filing method, time limits, evidence and outcome notification.
- A single denial **MAY expose both routes** when both apply (e.g. "this may be readable after step-up; if you contest the underlying decision, here is the appeal route"), **without leaking protected facts** — the routes are offered without revealing why a suppressed item is suppressed or whether a suppressed item exists.

## Alternatives considered

- **One index authorization covering both existence and payload** — rejected (R-14). Collapsing them makes lawful non-disclosure and step-up impossible without side channels: either everyone who can see the list can read everything, or hiding a payload also hides that access is even possible. Two decisions is the only way to get step-up + safe refusal.
- **Model appeal as "just re-authenticate" (step-up only)** — rejected (review #16). Re-authenticating does nothing for a person contesting a substantive `no-change` disposition; conversely an appeal body should not be invoked to fix a mere assurance gap. Conflating them denies people the correct remedy.
- **Treat correction as an in-place edit of the record** — rejected (invariant 7; ADR-0011; exchange-and-evidence.md). Correction is supersession or conspicuous statement-association, both append-only, so history and prior receipts stay valid and the contested state is auditable.
- **Adopt the CDR 10-business-day clock and dashboard rules as a compliance guarantee now** — rejected as a *claim* (legal-review file; R-12). The CDR/APP structure is implemented as a **candidate** profile (configurable response clocks, dashboard disclosures) so the capability exists and is testable, but asserting compliance requires the ADR-0015 corpus + human attestation. Implementing the mechanism is adopted; claiming compliance is gated.
- **Infer downstream correction from a single broadcast notification** — rejected (legal-review file; exchange-and-evidence.md). One notification does not correct N recipients' copies; each eligible recipient gets its own tracked duty with distinct states and lawful exceptions.

## Consequences

- **Positive:** people can discover, access, and correct records with signed acknowledgements and a governed disposition; non-disclosure and step-up work without side channels; appeal and step-up give the *right* remedy for the *right* problem; correction reaches the source system and prior recipients, not just the Databox annotation; everything is append-only durable evidence.
- **Negative / cost:** two-decision authorization, an existence-confidentiality model, a correction connector into source systems, per-recipient propagation duties, and configurable response clocks are substantial net-new build (DBX-14/DBX-23); the CDR/APP profile must be conspicuously labelled candidate until the legal gate clears, to avoid an unearned compliance claim.
- **Privacy & threat notes:** the central threat is the *existence side channel* — counts, URI patterns, event emissions, error-code or timing differences that reveal a suppressed record. The decision forbids all of these and reuses the 404-not-403 rule (DBX-01 §3). The index shows only authorized metadata and never other programs (README isolation). Staff identifiers in dispositions/audit are minimised in the consumer projection (exchange-and-evidence.md Audit). Prior-recipient notifications apply lawful exceptions and never leak protected facts.

## Failure behavior

Fail closed. If existence visibility for an item cannot be determined, the item is **omitted** and its existence is not signalled by any channel (default to non-disclosure, not disclosure). A payload request without sufficient assurance → denied with a step-up route where lawful, **without** confirming or denying existence beyond what the `existenceVisibility` rule permits; where existence itself is suppressed, return the indistinguishable 404-style response (no 403/401 that reveals presence). A correction request that does not durably commit → no acknowledgement is issued (no receipt before durable acceptance). A disposition or prior-recipient notification whose evidence is missing/unverifiable → treated as **not done**, never done-by-default (ADR-0012). Any compliance-labelled response MUST NOT be emitted while the legal gate is open.

- **CDR/APP compliance profile is a CANDIDATE, gated (not Blocked at the capability level).** The 10-business-day correction clock, rule-7.9/7.10 dashboard disclosures, fee prohibition, and any APP 10/12/13 timing/exceptions are implemented as **configurable candidate settings** and MUST NOT be asserted as compliance until the **ADR-0015 legal-compliance gate** clears (ingested corpus manifest + human attestation of the applicability/exception mapping, per R-12). **Unblocking input:** the attested compiled-policy bundle for the applicable CDR/Privacy-Act provisions. **Owning prompt:** the legal-policy mapping prompt via **DBX-07**, under ADR-0015. The *capability* (index, access, correction, appeal/step-up) is Adopted and proceeds on synthetic profiles now.
- The retained legal-mapping questions (which participant is the APP entity / CDR data holder, which records are personal information vs CDR data, which exceptions apply) are owned by the corpus workstream (legal-review file), not resolved here.
