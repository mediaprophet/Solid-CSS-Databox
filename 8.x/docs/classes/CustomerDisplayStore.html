    Class CustomerDisplayStore

    Persists a rendered customer-display playlist as an ordinary Solid resource through a CSS ResourceStore (see databox/solid-cms-plan.md, §10.3 / §1.6, real-time customer display).

    The portable renderer (CustomerDisplayRender) already produces the display's playlist as a plain, portable object; this store is the CSS-enhanced write leg that commits that playlist onto the normal Solid data path so it is WAC-governed, backed up with the pod, and — crucially — readable through plain LDP by any Solid client (the portable-core degradation, §1.4). The playlist object does not carry its own resolvable @id, so the caller names the canonical display IRI, which must live inside the configured storage space. The playlist is wrapped in a minimal JSON-LD @context so the persisted document round-trips as ordinary RDF rather than an invented dialect (the vanilla-Solid invariant); no term is renamed, the wrapper only maps the existing id to @id and the remaining camel-case keys onto the schema.org vocabulary.

    • persistState(
          displayIri: string,
          state: CustomerDisplayStateInput,
      ): Promise<PersistedCustomerDisplayResource>

      Write a lightweight display-state document at {displayIri}/state, representing the current display mode and active slide without requiring a full playlist re-render. This is the incremental update leg — the display client polls or subscribes to this resource to know "transaction view is active, slide X is showing", while the full playlist resource provides the complete deck when the display initializes or needs a refresh.
